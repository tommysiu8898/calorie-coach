## 🔍 Most likely causes for a crash-on-launch in an Expo/React Native app

### 1. **Missing or misconfigured native permissions / entitlements**
   Your app uses many native modules (HealthKit, Camera, Location, Notifications, etc.). If your `Info.plist` doesn’t include required usage description keys or entitlements, iOS may kill the app during launch.

   **Check your `ios/CaloriesCoach/Info.plist`** for keys like:
   - `NSHealthShareUsageDescription`
   - `NSHealthUpdateUsageDescription`
   - `NSCameraUsageDescription`
   - `NSLocationWhenInUseUsageDescription`
   - `NSPhotoLibraryUsageDescription`
   - `UIBackgroundModes` (if needed)

### 2. **Native module initialization failure**
   A native module (e.g., `expo-notifications`, `expo-location`, `RNAppleHealthKit`) might crash when trying to set up its delegate or request permissions during `AppDelegate` startup.

   **Action:** Open the app in Xcode and run it on a device/simulator. Look at the **console output** – it will show the exact exception and stack trace. Common culprits are:
   - HealthKit not available on simulator → check platform.
   - Missing `AppDelegate` method overrides for certain modules.

### 3. **Code signing / provisioning profile issue**
   Even if archive succeeds, the installed app may not be signed correctly for the target device.  
   **Check:** Are you installing a `Release` build on a physical device? Ensure the provisioning profile includes the device’s UDID and supports the required entitlements.

### 4. **Binary compatibility / Framework mismatch**
   Some CocoaPods frameworks (like `hermes-engine`, `React-Core-prebuilt`) are copied as XCFrameworks. If the slice for the target architecture (arm64) is missing or corrupt, the app will crash at `dyld`.

   **Action:** In Xcode, go to **Window → Devices and Simulators**, select your device, and view the device log. Look for `dyld: Library not loaded` or `Symbol not found` errors.

### 5. **EAS Update misconfiguration (rare but possible)**
   While the warning itself doesn’t crash the app, if your code relies on `expo-updates` and tries to fetch an update with an invalid URL/channel before the first render, it could cause an unhandled exception.  
   **Fix:** Run `eas update:configure` and set a default channel in `eas.json`. Or temporarily disable updates by setting `updates.enabled: false` in `app.json`.
