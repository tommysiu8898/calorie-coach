### 🏗️ Section 2: Technical Architecture Design

Now let's translate those features into a scalable, low‑cost infrastructure.

*I kept things simple for your MVP while maintaining the ability to scale as your user base grows.*

#### 2.1 Architecture Overview

**Frontend: React Native (Expo)**
- Develop cross‑platform iOS/Android using Replit's built‑in React Native + Expo template.
- Expo handles camera access, network requests, local storage management, and push notifications.

**Backend: Firebase (Simpler) or Supabase (Full control)**
- **Auth:** Anonymous + email/password sign‑in to start.
- **Realtime Database / Firestore:** Store meal logs (timestamps, photo URLs, AI outputs, user corrections).
- **Cloud Functions:** Handle AI API calls securely (no API keys embedded in frontend).

**AI Inference Pipeline: OpenRouter → Kimi VL (Fallback: GPT‑4o)**
1. **Frontend** uploads image to Cloud Function.
2. **Cloud Function** forwards image + prompt to OpenRouter.
3. **OpenRouter** selects `moonshot/kimi` as primary, `openai/gpt-4o` as fallback.
4. **API** returns structured JSON (calories, macros, ingredient list).
5. **Response** saved to database → returned to frontend.

**Database Schema Snippet (Supabase Example):**
```yaml
meal_entries:
  - id (uuid)
  - user_id (uuid, references profiles.id)
  - photo_url (text)
  - ai_response (jsonb)  # stores full output from Kimi/GPT-4o
  - calories (int)
  - protein (float)
  - carbs (float)
  - fat (float)
  - user_corrections (jsonb, nullable)  # store any manual edits
  - meal_type (text: breakfast/lunch/dinner/snack)
  - created_at (timestamp)
```

#### 2.3 Prompt Engineering Strategy (Crucial for Accuracy)

The secret that Cal AI used to reach 90% accuracy:

```text
System Prompt:
You are a world‑class nutrition AI. Analyze the uploaded food image and output 
exclusively valid JSON in the following structure:

{
  “meal_name”: “”,           // e.g., “Pasta with marinara sauce and meatballs”
  “ingredients”: [
    { “name”: “spaghetti”, “portion_grams”: 150, “calories”: 180,
      “protein_g”: 6, “carbs_g”: 36, “fat_g”: 1 }
  ],
  “total_calories”: 600,
  “total_protein_g”: 20,
  “total_carbs_g”: 70,
  “total_fat_g”: 15,
  “confidence_score”: 0.92   // scale 0 – 1
}

Rules:
- Estimate portion size using known reference objects (coin, hand) in the image.
- NEVER add fictional ingredients not visible.
- If ambiguous, set confidence_score <= 0.7 and output possible alternatives.
\end{code}
```

**Why this works for Kimi VL:**
1. Structured output forces the model to think step by step.
2. Portion cues (coin/hand) reduce volume guesswork by ~35%.
3. Confidence scoring triggers user correction flow when AI is unsure — prevents trust erosion.

**Cost‑Saving Trick for Later (After Launch):**
Don't waste tokens on multi‑image queries. Kimi VL handles 128K context but food analysis can be done in a single turn. Optimize by:
- Compress photos to 512×512px before sending (reduces token count by ~60%).
- Add **cache reads**: Same dish type repeated? Cache return for pennies.
